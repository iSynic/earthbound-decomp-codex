# EarthBound Decomp Encyclopedia

Local encyclopedia app for the EarthBound decomp knowledge base.

The app is intentionally split into two layers:

- `scripts/build-content.mjs` distills repo notes, source status dashboards, text-command manifests, source files, and tool listings into generated encyclopedia entries.
- `content/narrative-chapters.json` contains editable educational chapter drafts that are enriched with upstream evidence during generation.
- `public/` renders those entries with a ChatGPT-like shell: left table of contents, keyboard-friendly global search, document tabs, backlinks, and an evidence rail.
- Major index pages render as structured hubs, not raw note lists: narrative chapters, learning paths, topic areas, and source-bank entry points use card layouts with direct links into evidence.
- The generated catalog includes a `Catalog Build Status` page with build mode, provenance counts, and release-readiness signals.

## Generate Content

```powershell
npm run build:content
```

By default the builder runs in `local` mode and reads decomp material from the parent folder of `encyclopedia/`. To build the ROM-free authored release baseline:

```powershell
npm run build:content:authored
```

To build from the active decomp workspace instead:

```powershell
npm run build:content:codex
```

The same can be done with an explicit source root:

```powershell
node scripts/build-content.mjs --source-root "F:\Earthbound Decomp - Codex"
```

Generated files are written to:

```text
public/generated/catalog.json
public/generated/catalog.js
public/generated/source-snapshot.json
```

Validate internal entry links and generated reference links:

```powershell
npm run check:content
```

Check that the app package tree does not contain ROMs, raw extracted payloads, or generated binary media:

```powershell
npm run audit:release
```

Report editorial wording that may blur direct content with page-planning prose:

```powershell
npm run audit:editorial
```

The editorial audit is a report by default. It flags high/medium wording such as self-referential "this page should..." language and future-facing editor prose, plus low-severity "should/frontier" wording that may still be valid as a constraint.

Run the release baseline build, content check, and audit together:

```powershell
npm run prepare:release
```

For a release candidate, run:

```powershell
npm run release:check
```

The release check must leave `public/generated/` in authored mode. Do not package or publish a local/Codex catalog build; local builds can include generated source, asset, routine, symbol, and manifest entries that belong only after a user provides a ROM.

## Workspace Split Audit

Before removing decomp material from `F:\Earthbound Decomp - Encyclopedia`, run the read-only split audit:

```powershell
npm run audit:workspace-split
```

The audit compares the app workspace root against `F:\Earthbound Decomp - Codex`, classifies top-level material by release policy, and calls out files that are missing or different in the reference workspace. Treat any `missing-in-reference` or `differs-from-reference` row as a cleanup blocker until it is archived, copied to the reference workspace, or intentionally discarded.

To compare against a different reference checkout:

```powershell
node scripts/audit-workspace-split.mjs --reference-root "F:\Earthbound Decomp - Codex"
```

## Release Artifact Policy

The release boundary is:

- Packaged: Electron shell, curated docs, learning paths, source-safe ROM knowledge, public payload-free manifests, validation logic, and the authored catalog.
- User-provided: ROM file and optional local decomp/source workspace paths.
- Generated locally: `src/`, some `asset-manifests/`, renderer fixtures, source snapshots, search catalog, extracted references, preview sheets, SPC/WAV exports, and cache files.
- Never distributed: ROM bytes, raw extracted copyrighted assets, generated audio/images that contain ROM-derived content, or generated byte-equivalent source built from the user's ROM.

The packaged app must remain useful in authored mode without a ROM. It must not include source files, generated source indexes, asset manifests, raw extracted assets, ROM files, audio renders, image renders, or cache files. ROM selection and local workspace generation extend the app after first run, with generated artifacts written under the user's app-data workspace.

## Packaging

Install dependencies before packaging:

```powershell
npm install
```

Build a Windows release candidate:

```powershell
npm run dist:win
```

Build a Linux AppImage release candidate:

```powershell
npm run dist:linux
```

Build a Linux ZIP fallback from Windows:

```powershell
npm run dist:linux:zip
```

Linux AppImage builds are best produced and smoke-tested on Linux or CI. Cross-building AppImage from Windows may fail when app-builder needs Linux-style symlinks; the ZIP target is a useful fallback, but it still needs a real Linux launch test before release.

Before publishing binaries:

- run `npm run release:check`;
- inspect the package contents for accidental ROM/source/asset payloads;
- add a root `LICENSE` file with the chosen project license;
- include the notice guidance from `RELEASE-LICENSES.md`;
- test first-run notes-only mode and ROM-backed generation on a clean user-data directory.

## Launch

After installing dependencies:

```powershell
npm install
npm start
```

`npm start` opens the already-built catalog. For development, rebuild first and then launch:

```powershell
npm run dev
```

To rebuild from `F:\Earthbound Decomp - Codex` and launch:

```powershell
npm run dev:codex
```

`npm run start:codex` is kept as a compatibility alias for `npm run dev:codex`. The viewer can also be inspected directly by opening `public/index.html` after running a content build.

## Local ROM Workspace

The Electron app exposes a `Local Workspace` flow for ROM-backed generated content:

- first run can be skipped for notes/contracts-only browsing;
- `Add ROM` verifies headered/headerless EarthBound ROM size and SHA-1;
- a verified ROM creates a hash-keyed app-data workspace manifest;
- generated source, assets, audio, cache files, and exports stay local-only under app data;
- the `Asset Library` page is the organized browsing/export surface for those generated families;
- ZIP exports are user-local and may contain ROM-derived payloads, so they are never app release artifacts.

The static browser view can verify a selected ROM with Web Crypto, but filesystem generation requires Electron.

## Current Scope

The first catalog includes:

- Curated orientation chapters.
- Generated bank entries for all `C0..EF` banks.
- Generated top-level text-command entries from `notes/text-command-semantics-manifest.md`.
- Evidence-note entries for local notes.
- Routine/module entries for source-heavy bank files in `C0`, `C1`, `C2`, `C4`, and `EF`.
- Asset manifest and asset entries from `asset-manifests/`.
- Workflow, tool, and source index pages.
- Upstream source status and changed-file snapshot pages.
- Narrative chapter entries for active decomp workstreams.
- An authored C3/script chapter cluster covering the closed byte layer, native callback contracts, callback argument semantics, movement values, direction/facing values, animation templates, and source-emission model.
- An authored text/localization chapter cluster covering the text VM, command-family model, recovered `.MSG` authoring format, and macro lowering.
- An authored table-contract chapter cluster covering runtime row contracts, D5 split rows and field families, battle/enemy tables, PSI menu and ability tables, PSI name/display rows, item/equipment tables, condiment rows, service/destination rows, shop inventory rows, character growth tables, route/support tables, and map object tables.
- An authored asset/data chapter cluster covering the contract model, battle visual assets, overworld object/sprite contracts, map scene and substrate contracts, and audio pack stream/snapshot/backend boundaries.
- Structured hub views for narrative chapters, topics, guided learning paths, and source-bank browsing.

The next useful step is to promote more generated evidence into curated chapters: NPC AI bytes, concrete shop selector paths, C3 encoded PSI menu row decoding, map preview/editor rules, audio runtime state capture, and concrete editor-facing rules for the asset families that are already structurally closed.
