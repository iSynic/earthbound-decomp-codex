# C3 build-candidate source conventions

This note defines the first C3 build-candidate gate. It is intentionally narrower
than a full assembler build: a module can become a build candidate when its
source is ready for assembler integration and its ROM range is protected by a
repeatable byte-range/signature harness.

## Source Dialect

The C3 prototypes use a ca65-like 65816 dialect:

- one address-prefixed public label per known entry, e.g.
  `C3F1EC_TryRepairJeffBrokenInventoryItem:`
- internal branch labels keep the address prefix when the original address is
  known
- constants are declared with `NAME = $VALUE`
- ordinary local state may stay as direct-page operands (`$02`, `$04`, etc.)
  when the source contract already documents the local meaning
- external routines may remain raw long calls until the cross-bank symbol table
  is generated; the contract comment beside the call family is the named source
  dependency

This convention is assembler-oriented, but these files are not linked into a ROM
yet. The build-candidate gate is the point where syntax, labels, and bytes are
stable enough to feed an assembler integration task.

## Build-Candidate Requirements

A C3 helper module can be marked `build-candidate` when:

- its planned source range is closed
- all expected source-emission labels are present
- `tools/validate_c3_source_signatures.py --strict` passes for the module
- `tools/validate_c3_build_candidate_ranges.py --strict` passes for the module
- a byte-range manifest records the ROM range, size, SHA-1, and first/last bytes
- remaining non-assembler work is explicitly listed as follow-up hardening, not
  hidden in the source body

## Assembler-Ready Pilot Requirements

An individual build-candidate module can be marked `Assembler contract:
pilot-ready` when:

- every symbolic operand resolves to a same-file constant or label
- every external control-flow target is named by a same-file absolute alias, not
  a raw six-digit literal operand
- local branch labels remain address-prefixed so the source signature validator
  can continue to anchor them to ROM addresses
- the module still passes the source signature and build-candidate range gates

This pilot gate is still short of a full linked build. It proves that one source
file has moved from "documented annotated source" to "assembler-shaped source
with byte guards still intact."

## Byte-Equivalence Pilot Requirements

The reusable gate is `tools/validate_source_bank_byte_equivalence.py`; the
C3-specific `tools/validate_c3_byte_equivalence.py` wrapper remains available
for compatibility. The validator translates the ca65-like source into scratch
Asar form, patches a clean ROM copy at the module's original `org`, and compares
the protected range against the original ROM bytes. The same tool can also run a
combined helper scaffold with `--combined`, assembling all selected modules into
one generated patch and one ROM copy before comparing each protected range.
Durable source-bank scaffolds are generated by
`tools/build_source_bank_scaffold.py`; for C3 the checked-in Asar scaffold is
`src/c3/bank_c3_helpers_asar.asm`.

A module passes this gate when:

- the scratch Asar translation assembles without errors
- source-adjacent data gaps from `build/c3-build-candidate-ranges.json` are
  preserved as raw bytes in the scratch assembly
- every byte in the protected module range matches the original ROM
- for the combined scaffold, all selected modules share one generated Asar file
  and one patched ROM while still matching every protected range

  `tools/validate_source_bank_byte_equivalence.py --bank C3 --module all --combined --scaffold src/c3/bank_c3_helpers_asar.asm --strict`

These candidates are not a claim that C3 as a whole is ready to link as a full
bank. They prove the next useful invariant: source text, source-emission
metadata, assembler-shaped operands, scratch assembly, and ROM bytes agree for
closed helper modules under repeatable gates.
