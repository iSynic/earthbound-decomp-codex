## Machine-readable manifest

`build/working-names-c0-c2.json` is generated by:

```powershell
python tools\build_working_name_manifest.py --banks C0 C1 C2 --output build\working-names-c0-c2.json
```

The source-label artifacts are generated by:

```powershell
python tools\emit_source_labels.py --manifest build\working-names-c0-c2.json --output-dir build\labels --banks C0 C1 C2
```

The struct/data contract artifacts are generated by:

```powershell
python tools\build_data_contract_manifest.py
```

Validate the manifest with:

```powershell
python tools\validate_data_contracts.py
```

Look up contract-backed addresses and fields with:

```powershell
python tools\lookup_data_contract.py D5:9589+0x46 "BATTLERS_TABLE[3].afflictions" '$ADD4+0x61'
```

Use a contract directly in the generic ROM table inspector with:

```powershell
python tools\inspect_table.py --contract ENEMY_CONFIGURATION_TABLE --index 0 --count 1 --field actions:w:0x46
```

To find the next notes with lots of address mentions but few explicit working-name bullets:

```powershell
python tools\audit_working_name_coverage.py --banks C0 C1 C2 --min-missing 20
```

To instead find notes that still mention addresses with no promoted name anywhere in the corpus:

```powershell
python tools\audit_working_name_coverage.py --banks C0 C1 C2 --min-missing 20 --scope global
```

For a cleaner pass over likely routine headings and note titles, ignoring most incidental cross-references:

```powershell
python tools\audit_working_name_coverage.py --banks C0 C1 C2 --scope global --mention-kind headings --min-missing 1
```

Schema summary:

- `schema`: manifest version string
- `banks`: included banks
- `summary.entries`: total promoted working-name entries
- `summary.by_bank`: entry count per bank
- `summary.by_tag`: inferred subsystem tags
- `entries[].address`: CPU address, normalized as `C?:XXXX`
- `entries[].name`: proposed local symbol name
- `entries[].status`: currently `working-name`
- `entries[].confidence`: `proposed` or `corroborated`
- `entries[].tags`: inferred subsystem tags from note/name context
- `entries[].evidence`: note path, line, and section for each proposal source

Schema summary:

- `schema`: contract manifest version string
- `summary.contracts`: total contract count
- `summary.fields`: total field count
- `summary.by_domain`: contract count by `wram-root`, `wram-overlay`, or `rom-table`

- `contracts[].stride`: record stride or struct size
- `contracts[].count`: fixed count when known
- `contracts[].struct`: reference struct name
- `contracts[].confidence`: current confidence tier
- `contracts[].evidence`: source notes and reference files
- `contracts[].fields[]`: named offsets, sizes, counts, notes, and element names where applicable

## Cross-bank bridge points

High-value C0-C2 interfaces:

- C0 upload and palette helpers are consumed heavily by C2 battle backgrounds and PSI/sprite palettes.
- C1 text/window routines call C2 HP/PP window helpers for tile and roller state.
- C1 battle UI routines feed C2 battle action selection and targetting.
- C2 battle visuals call into C3/C4 data/script assets for PSI frames, palettes, overlay script pointers, and special visual routines.
- C0 overworld transition and C1 text command families both queue or trigger C4/E F script paths; those are natural seams once C3 starts.

## Promotion policy

Use this ordering before writing names into source or a port model:

1. Prefer names with explicit manifest entries.
2. If no manifest entry exists, prefer a focused note with a `Working Names` section and add one before promoting.
3. If a note only has prose, do not promote the name blindly; first add an explicit bullet and rerun `tools/build_working_name_manifest.py`.
4. Keep reference labels from `refs/` as corroboration, not authority, unless the local byte/caller evidence agrees.
5. Preserve uncertainty in names for pointer-only, data-only, or direct-call-missing entries.
